# Need

## 背景

上一轮已经把正式 project 执行空间收敛到群聊，但发起人进一步明确：项目的管理动作应放在单聊里完成，包括管理 project、创建群和建立绑定关系。

这意味着 PoCo 不能只回答“project 应该在哪个群里跑”，还要回答“用户应当在哪里管理这些 project”。

## 需求信号

- 用户希望在单聊里集中管理 project
- 用户希望由单聊触发建群和 project 绑定，而不是手工先建好群再配置
- 用户希望把“管理动作”和“执行动作”分开，避免单聊被执行噪声污染

## 来源

- 发起人在交互模型讨论中的直接要求

## 场景

- 在单聊里创建新 project
- 在单聊里为 project 创建或绑定一个飞书群
- 在群里继续该 project 的任务执行、状态查看和确认动作

## 频率/影响

该需求会改变正式交互模型的入口分工。若不明确单聊与群聊的职责分离，PoCo 后续 project 生命周期、群绑定和 session 路由都会继续模糊。

## 备注

这轮讨论的重点是交互责任分工，不是立即定义完整 project 管理后台。
