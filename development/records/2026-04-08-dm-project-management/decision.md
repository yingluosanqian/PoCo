# Decision

## 待选问题/方案

- 方案 A：单聊和群聊都承载 project 管理与执行
- 方案 B：单聊负责 project 管理，群聊负责 project 执行
- 方案 C：只保留群聊，不定义单聊管理入口

## 当前决策

采用方案 B。

PoCo 的交互模型调整为：

- `DM -> control plane`
- `Group -> project workspace`

用户在单聊中管理 project，包括创建、查看、建群和绑定；正式的 task 执行、状态回推和确认动作主要发生在 project 群中。

## 为什么这样选

- 这符合发起人刚刚明确的使用方式
- 单聊天然适合低频、管理类操作
- 群聊更适合承载持续执行、多人可见和结果沉淀
- 这样既保留了群作为 project 执行容器的清晰边界，也避免把 project 管理动作塞进群里

## 为什么不选其他方案

- 不选方案 A：会让管理动作和执行噪声重新混在一起
- 不选方案 C：虽然群执行边界清楚，但用户仍缺一个自然的 project 管理入口

## 风险

- 单聊与群聊之间的 project 绑定流程需要额外定义
- DM control plane 可能逐步长出较多管理命令，需要保持克制
- 如果用户坚持在单聊直接执行正式任务，后续需要定义例外策略

## 后续影响

- project lifecycle 应先围绕 DM 入口设计
- group binding 将成为 project 创建流程的一部分
- 正式执行路径应继续保持在群中，不回退到单聊主执行模型
