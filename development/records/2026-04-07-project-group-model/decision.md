# Decision

## 待选问题/方案

- 方案 A：所有 project 共用一个 bot 聊天框，通过命令切换当前 project
- 方案 B：一个 project 对应一个飞书群，群作为正式 project 容器
- 方案 C：一开始就做群、单聊、卡片、多入口并存的复杂模型

## 当前决策

采用方案 B。

PoCo 的正式交互主路径采用“一个 project 一个飞书群”的模型。群绑定 project，群内命令默认落到该 project。session 继续存在于 project 之下，而不是直接把群等同于 session。

## 为什么这样选

- 这最符合发起人对“不要混在一个聊天框里”的明确要求
- 群天然提供了 project 边界，能显著降低手机端认知负担
- 群作为 project 容器后，`/run`、`/approve`、`/status`、通知回推都能获得稳定默认归属
- 这比在单个 bot 会话中维护 project 切换状态更直接，也更适合 MVP 收敛

## 为什么不选其他方案

- 不选方案 A：虽然实现上可能更省，但会把 project 归属压力转嫁给用户记忆和命令切换，容易混乱
- 不选方案 C：当前阶段过早引入多入口并存模型，会显著增加范围和交互复杂度

## 风险

- 一个群绑定一个 project 会提高 project 初始化成本
- 若用户希望在单聊中完成正式 project 操作，后续需要明确例外路径
- 群成员与 project 成员边界未来可能需要权限模型补强

## 后续影响

- `project` 将成为群级归属对象
- `session` 需要重新明确为 project 下的连续工作流，而不是聊天容器本身
- 单聊在 MVP 中应重新定位为调试入口、个人入口或待后续定义的辅助入口
