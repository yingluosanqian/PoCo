# Problem

## 背景

PoCo 需要在手机端承载持续的远端 agent 工作流，而不是只触发一次性命令。

当前系统虽然已有飞书消息入口和任务执行链路，但还没有定义 project 应如何映射到消息交互空间。若默认把多个 project 放进同一个 bot 聊天框，产品连续性和交互清晰度都会迅速恶化。

## 相关需求

- 不同 project 之间有清晰边界
- 用户在手机上能快速知道当前命令属于哪个 project
- `/approve`、`/status`、结果回推等操作不会跨 project 混淆

## 当前状态

- 当前 PoCo 的正式交互模型尚未定义 project 容器
- 当前飞书网关只按消息事件和任务响应工作，没有 project 级归属模型
- 当前若把多个 project 混在同一个聊天框中，用户需要依赖额外心智记忆来区分上下文

## 问题定义

PoCo 当前缺少一个足够清晰的 project 级消息容器模型，导致多 project 并存时，移动端交互容易出现上下文归属不清、确认误操作和结果污染。

## 为什么这是个真实问题

- 这会直接削弱手机端交互的清晰度，而小屏场景下这种混乱成本更高
- 这会影响刚刚建立的 `session/handoff` 设计，因为 session 需要上层归属对象
- 如果不先收敛 project 容器，后续状态展示和确认指令都难以形成稳定默认行为

## 不是什么问题

- 不是“群聊是否比单聊技术上更容易接入”的问题
- 不是“是否先做完整权限协作系统”的问题
- 不是“马上支持复杂的 project 切换命令集”的问题

## 证据

- 发起人明确指出“不同的 project 被放到同一个 bot 聊天框，会很混乱”
- 发起人进一步明确表示“可以直接往一个 project 一个群的思路做”
- 当前系统尚未定义 project 容器，属于真实缺口，而不是已存在但体验欠佳的能力
