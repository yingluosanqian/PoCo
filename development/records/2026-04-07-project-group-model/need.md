# Need

## 背景

PoCo 当前已经有消息入口和任务链路，但发起人明确指出：如果把不同 project 都放进同一个 bot 聊天框，用户交互会非常混乱。

随后发起人进一步收敛出偏好：正式使用路径可以直接采用“一个 project 一个群”的方式。

## 需求信号

- 不同 project 的任务、进展、确认和结果不应混在同一个聊天上下文里
- 用户希望用群天然区分 project，而不是在同一个聊天框里频繁切换 project
- 正式使用路径应优先服务清晰性，而不是极致灵活性

## 来源

- 发起人在讨论用户交互方式时的直接决策

## 场景

- 用户同时维护多个 project，需要在手机上区分不同工作流
- 群内多次触发 `/run`、`/approve`、`/status` 等命令时，希望默认归属明确
- 多人围绕同一个 project 查看进展和确认动作时，希望边界稳定

## 频率/影响

该需求会直接影响 PoCo 的正式用户交互模型。若不先明确 project 的承载方式，后续 session、status、approval 和通知模型都会持续混乱。

## 备注

这条需求讨论的是正式交互主路径，不等于要求立即放弃单聊入口。单聊仍可作为调试或个人入口另行定义。
