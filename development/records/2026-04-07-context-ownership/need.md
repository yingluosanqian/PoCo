# Need

## 背景

PoCo 当前已经能通过飞书和本地 demo 入口触发远端 agent 任务，但用户在实际联调中指出：系统看起来还无法稳定维护“上下文”。

这个信号不是单纯的实现抱怨，而是对产品连续性能力的直接质疑。用户进一步提出一个关键问题：这些上下文是否本应由 Codex 维护，而不是由 PoCo 维护。

## 需求信号

- 用户希望在手机上中断后还能继续推进同一条工作流
- 用户希望系统理解“继续刚才那个”“按上次方案改一下”这类连续指令
- 用户希望上下文责任边界清楚，不希望 PoCo 无限制复制 agent 内部记忆
- 用户明确要求先按开发规范分析，而不是直接实现

## 来源

- 发起人在真实联调中的直接反馈

## 场景

- 在飞书单聊中连续推动同一项开发工作
- 任务完成后，希望后续命令能自然继承最近结论
- agent 中断、重启或切换后，希望手机端仍知道“当前工作流进行到哪里”

## 频率/影响

该需求与项目目的直接相关。若 PoCo 无法提供连续性交接能力，移动端交互就会退化成一次性任务触发器，难以支撑“碎片化打断后恢复”的真实使用方式。

## 备注

这条需求不是要求 PoCo 立即实现完整 memory 系统，而是要求先厘清上下文责任边界。
