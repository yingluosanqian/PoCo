# Problem

## 背景

PoCo 的目标不是单次触发远端任务，而是让开发者在移动场景下持续控制服务器侧 AI agent。

当前 MVP 已经有消息入口、任务状态和结果回推，但还没有定义“连续性交接上下文”应由谁持有，也没有定义用户在中断后恢复工作流时系统应依赖什么对象。

## 相关需求

- 在手机上恢复刚才的工作流
- 支持“继续”“按上次结论修改”这类连续指令
- 明确 PoCo 与 Codex 等执行器之间的上下文责任边界

## 当前状态

- 当前 PoCo 只维护 task state，不维护 session continuity
- 当前 Codex adapter 只负责单次执行，不对产品层暴露稳定的连续性语义
- 当前系统无法稳定回答“继续什么”“基于哪段历史继续”
- 当前还没有关于上下文归属的明确决策记录

## 问题定义

PoCo 当前缺少一层独立于消息平台、也独立于具体 agent backend 的“产品级连续性交接上下文”，导致用户在移动场景下无法低摩擦恢复和延续同一条远端 agent 工作流。

## 为什么这是个真实问题

- 这直接影响项目目的中“碎片化打断后恢复上下文”的承诺是否成立
- 如果完全依赖 backend 内部上下文，PoCo 就无法向用户提供稳定的恢复对象和状态解释
- 如果完全由 PoCo 重建全部上下文，又会过早引入重型 memory 系统，违反当前复杂度约束

## 不是什么问题

- 不是“Codex 是否支持更长上下文窗口”的问题
- 不是“先做一个完整 session memory 平台”的问题
- 不是“飞书聊天记录要不要全部持久化”的问题
- 不是“先让 PoCo 完整复制 agent 内部记忆”的问题

## 证据

- `purpose` 已明确要求支持查看上下文、阻塞点和碎片化恢复
- `constraints` 已明确要求交互必须允许中断后恢复上下文
- 当前 `state` 已表明系统仅有 task state，没有 session continuity
- 发起人在真实联调中直接指出“看起来没办法维护上下文”
