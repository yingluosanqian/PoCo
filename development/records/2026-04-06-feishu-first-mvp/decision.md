# Decision

## 待选问题/方案

- 第一阶段优先接飞书 bot
- 第一阶段优先接 Slack bot
- 第一阶段同时支持飞书和 Slack
- 暂不选具体消息入口，先做更抽象的平台层

## 当前决策

第一阶段优先接飞书 bot，作为 PoCo 第一条可运行主链路的消息入口。

当前实现栈使用 Python。

当前 agent 支持规划包含 Codex、Claude Code、Cursor Agent，其中第一优先级是 Codex。

## 为什么这样选

- 发起人已明确表达“优先接飞书”的当前方向
- 发起人已明确表达 agent 方向中优先支持 Codex
- 先收敛到单一平台，能显著降低接入、鉴权、交互和调试复杂度
- 当前项目还处于定义早期，先跑通一条完整主链路比同时兼容多个入口更重要
- 这符合当前约束：先保证真实可用、先保证单条主链路跑通、避免为未来平台扩展过早做重型抽象

## 为什么不选其他方案

- 不先选 Slack：当前没有来自产品目标或资源条件的额外理由说明 Slack 比飞书更优先
- 不同时支持飞书和 Slack：会过早引入平台兼容层、接口抽象和测试矩阵，扩大范围
- 不先做抽象平台层：在缺少第一条已验证主链路前，抽象很容易脱离真实问题并抬高复杂度
- 不先同时实现 Claude Code 和 Cursor Agent：会把执行器适配、状态差异和错误处理复杂度提前拉高

## 风险

- 飞书优先可能让后续 Slack 接入时暴露平台耦合问题
- 若早期设计过度贴合飞书交互模型，后续平台迁移成本会上升
- 如果飞书平台能力与目标工作流不匹配，可能需要回调本轮决策

## 后续影响

- 本轮计划应围绕飞书单平台 MVP 收敛范围
- 核心能力仍应尽量平台无关，但不为了“以后扩展”先做重型抽象
- 进入设计时，需要重点约束消息入口层与核心 agent 控制层的边界
- 实现层优先搭建 Python 服务端骨架，并以 HTTP webhook 方式承接飞书入口
- 执行器层应形成 agent adapter 边界，但当前只把 Codex 做成真实实现
