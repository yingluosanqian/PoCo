# Design

## 设计目标

在“飞书优先、单平台 MVP”的范围内，设计一条最小但完整的 PoCo 主链路，使用户能够通过手机消息入口驱动服务器侧 AI agent，并看到进展、处理确认、接收结果。

设计必须满足以下要求：

- 飞书接入优先，但不把飞书细节扩散到核心任务控制逻辑
- 核心执行留在服务器侧
- 长任务具备状态可见性
- 人工确认节点具备明确闭环

## 核心抽象

### 1. Message Gateway

负责接收和发送飞书消息事件。

职责：

- 接收飞书 webhook / 回调事件
- 接收飞书 long connection 消息事件
- 验证来源与基础身份
- 把平台事件转换为内部统一交互请求
- 把内部状态更新转换为飞书消息输出

边界：

- 只处理平台协议和轻量适配
- 不负责任务编排、agent 执行或业务决策

补充说明：

- 本地 demo HTTP 入口可以作为联调辅助入口存在，但不改变“飞书是正式优先入口”的产品决策
- 传输模式可以是 webhook 或 long connection，但它们都必须复用同一个 `FeishuGateway`，避免分叉业务逻辑

### 2. Interaction Service

负责解释用户在消息入口中的意图，并把它们路由为系统动作。

职责：

- 解析“发起任务”“查看状态”“确认/拒绝”等交互
- 维护与单个任务相关的会话上下文
- 调用任务控制层创建或推进任务

边界：

- 不直接执行 agent
- 不持有平台特定逻辑以外的复杂状态

### 3. Task Controller

负责任务生命周期管理，是第一阶段的核心业务边界。

职责：

- 创建任务
- 推进任务状态
- 记录关键事件
- 决定何时向用户回传状态
- 决定何时进入“等待人工确认”

边界：

- 不依赖飞书具体协议
- 不直接承担底层 agent 进程实现细节

补充约束：

- Task Controller 负责状态变更，不直接承担后台线程调度
- 后台调度应由独立调度层驱动，以避免把执行模型耦合进核心任务状态逻辑

### 4. Agent Runner

负责在服务器侧实际驱动 AI agent 执行任务。

职责：

- 接收任务控制层下发的任务
- 启动、继续、暂停或终止 agent 执行
- 回传执行中的关键事件、阻塞点和最终结果

边界：

- 对上暴露任务级接口，而不是裸 shell 透传
- 不负责面向用户的消息交互

第一阶段取向：

- 建立统一的 agent runner 接口
- 优先实现 Codex 适配
- 为 Claude Code、Cursor Agent 预留后续扩展位置，但当前不实现假适配

### 5. Task State Store

负责保存 MVP 所需的最小任务状态。

至少应记录：

- 任务标识
- 发起人身份
- 当前状态
- 最近关键事件
- 是否等待人工确认
- 最终结果摘要

### 6. Approval Checkpoint

负责处理需要用户确认的关键节点。

职责：

- 将任务切换为等待确认状态
- 生成可回到该任务的确认动作
- 接收用户确认或拒绝
- 将结果回传给 Task Controller

### 7. Background Dispatcher

负责把任务执行从 webhook 请求线程中解耦出来。

职责：

- 在任务创建后异步触发执行
- 在批准后异步恢复执行
- 捕获执行异常并回写任务失败状态
- 在关键状态变化后触发通知

边界：

- 不定义业务状态，只消费和推进 Task Controller 已批准的状态流
- 不承担平台协议细节

### 8. Task Notifier

负责把关键任务状态变化主动推送回消息入口。

职责：

- 在等待确认时主动推送任务状态和操作提示
- 在完成或失败时主动推送结果摘要
- 基于任务保存的回复目标选择回推位置

边界：

- 只处理通知，不参与任务调度和执行
- 当前可由飞书实现，但接口应保持可替换

## 模块/接口影响

第一阶段建议形成如下模块边界：

- `platform/feishu`
  处理飞书事件接收、签名校验、消息发送、交互动作映射
- `demo`
  处理本地 HTTP 联调入口，把请求映射为与消息入口一致的内部交互动作
- `interaction`
  处理用户意图解析和交互路由
- `task`
  处理任务模型、状态流转、后台调度和关键事件
- `agent`
  处理服务器侧 agent 驱动适配与执行器选择
- `storage`
  处理任务状态与事件的最小持久化

对外主要接口形态应是：

- 来自飞书的“事件输入接口”
- 来自本地 demo 的“命令输入接口”
- 面向内部的“创建任务 / 查询任务 / 确认任务”接口
- 来自 Agent Runner 的“事件回传接口”
- 面向飞书的“状态通知输出接口”
- 面向后台调度器的“异步启动 / 异步恢复”接口

## 状态变化

第一阶段的最小任务状态建议为：

- `created`
- `running`
- `waiting_for_confirmation`
- `completed`
- `failed`
- `cancelled`

最小状态流：

1. 用户在飞书中发起任务，任务进入 `created`
2. Background Dispatcher 异步触发执行，任务进入 `running`
3. 若需要人工确认，任务进入 `waiting_for_confirmation`
4. Task Notifier 主动回推确认提示
5. 用户确认后由 Background Dispatcher 异步恢复执行并回到 `running`，或拒绝后进入 `cancelled`
6. Task Notifier 在终态时主动回推结果
7. 执行成功进入 `completed`，失败进入 `failed`

## 方案比较

### 方案 A：飞书平台逻辑与任务控制完全耦合

优点：

- 初期实现看似更快

缺点：

- 平台细节会污染核心逻辑
- 后续接入 Slack 时返工概率高
- 难以保持任务状态模型清晰

### 方案 B：先做完整多平台抽象层，再接飞书

优点：

- 表面上扩展性更强

缺点：

- 在没有真实主链路前容易过度抽象
- 会显著增加当前复杂度和实现成本

### 方案 C：飞书薄适配 + 平台无关的任务控制核心

优点：

- 既能快速收敛到飞书 MVP
- 又能把平台耦合限制在入口层
- 更符合当前约束和决策

缺点：

- 仍需谨慎避免把飞书交互模型渗透进核心任务状态

## 最终方案

采用方案 C：

以飞书作为唯一优先入口，但将飞书相关协议、事件和消息格式限制在 `Message Gateway` 层；服务器侧围绕 `Task Controller + Agent Runner + Task State Store + Approval Checkpoint` 形成平台无关的最小核心。

这样可以先跑通一条真实主链路，同时保留后续增加其他平台入口的可能性。

同时，通过引入 Background Dispatcher 和 Task Notifier，可以避免 webhook 长时间阻塞，并让移动端真正收到异步结果回推。

## 风险与兼容性

- 若飞书交互能力不足以承载确认动作，可能需要调整消息模型
- 若 Task Controller 过度依赖某种 agent 运行方式，后续替换执行器会变难
- 若状态模型过粗，移动端上下文会不清晰；若状态模型过细，MVP 会复杂化
- 第一阶段默认以单用户、单主链路为前提，后续扩展到多用户协作时需要重新审视权限和冲突控制
