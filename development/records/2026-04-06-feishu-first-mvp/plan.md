# Plan

## 目标

定义 PoCo 第一阶段的飞书优先 MVP 范围，使项目进入可执行的首轮产品实现准备状态。

## 范围

- 以飞书 bot 作为唯一优先消息入口
- 定义一条最小但完整的用户主链路：发起任务、查看进展、处理确认、接收结果
- 约束系统角色边界：手机端消息入口、服务器侧 agent 执行、状态回传与确认节点
- 为后续实现准备最小设计输入，而不是直接铺开所有平台或功能
- 将任务执行从 webhook 请求线程中解耦为后台执行
- 在等待确认和终态时主动回推消息
- 提供本地 HTTP demo 入口，降低联调成本并支持脱离飞书验证整条链路

## 不在范围内的内容

- 同时支持 Slack 或其他消息平台
- 团队协作、多用户、多租户能力
- 通用 bot 平台或开放插件系统
- 完整 IDE 式移动编辑体验
- 结算、配额、复杂权限中心等扩展体系
- 把本地 demo 入口演化成第二套正式产品入口

## 风险点

- 容易把“飞书接入”误写成全部问题，从而忽略任务模型和状态模型
- 容易在尚未明确工作流前直接写接入代码，导致后续返工
- 飞书平台交互能力可能限制消息结构和确认方式
- 安全边界如果定义不足，早期接入可能默认暴露高风险能力
- 后台执行若没有明确状态回写和通知边界，容易出现“任务已经完成但手机端无感知”的断链

## 验收标准

- 已明确第一阶段为什么只选飞书而不选其他平台
- 已明确第一阶段用户主链路包含哪些动作
- 已明确本轮不做什么，能排除范围蔓延
- 已为下一步设计或实现提供清晰边界

## 实施顺序

1. 固化 `need/problem/decision/plan` 记录
2. 补充稳定层中的当前问题、当前决策、当前计划摘要
3. 基于飞书优先 MVP 形成第一版设计
4. 设计通过后，再进入首轮实现
