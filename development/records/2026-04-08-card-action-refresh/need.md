# Need

## 背景

卡片最小信息架构已经收敛出来，但还缺两块真正决定实现复杂度的内容：

- 卡片 action 如何映射到后端意图
- 卡片在状态变化后如何刷新

如果这两块不先定义，后续实现很容易出现 action id 混乱、回调职责漂移，以及卡片更新策略不一致。

## 需求信号

- 需要继续推进卡片实现前设计
- 需要把卡片动作从“按钮文案”收敛成稳定的后端意图
- 需要定义卡片刷新和状态同步策略

## 来源

- 当前最小 IA 已经落成，残留问题明确指向 action 映射与刷新策略

## 场景

- 用户点击 `Create Group`、`Open Workspace`、`Approve`、`Continue`
- 任务状态从 running 进入 waiting / completed / failed
- 卡片需要决定是整体替换、局部刷新，还是只回发一张新卡

## 频率/影响

这一步会直接决定卡片交互实现的稳定性，是进入实现前的必要设计。

## 备注

本轮仍然不定义飞书 schema 细节，而是定义后端协议和刷新原则。
