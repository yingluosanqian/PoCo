# Problem

## 背景

PoCo 已经有 card-first 方向和最小信息架构，但仍没有定义卡片动作如何映射到系统意图，也没有定义状态变化后卡片应如何更新。

缺少这两块时，卡片实现会陷入两个典型问题：

- action id 只是零散按钮，而不是稳定意图协议
- 不同卡片和不同状态变化采用不同刷新方式，导致行为难以预测

## 相关需求

- 稳定的卡片 action 语义
- 可预测的卡片刷新和状态同步方式
- DM 与群内卡片都遵守同一套后端交互原则

## 当前状态

- 当前 records 已定义卡片类型和动作名称，但未定义 action id 协议
- 当前也未定义“哪些动作更新原卡片，哪些动作新发卡片，哪些动作只回执”
- 当前实现层还没有任何卡片 action handler

## 问题定义

PoCo 当前缺少一套统一的“卡片动作意图协议 + 卡片刷新策略”，导致 card-first 方向还无法转成可实现、可维护的后端交互模型。

## 为什么这是个真实问题

- 没有统一 action 协议，后续卡片会把业务语义散落在按钮文案和临时字段里
- 没有刷新策略，用户会看到不可预测的卡片行为
- DM 和群卡片如果各自发明规则，后续维护成本会迅速增加

## 不是什么问题

- 不是按钮命名风格问题
- 不是视觉微交互问题
- 不是立刻实现卡片回调代码的问题

## 证据

- 上一轮 record 的残留问题已明确指出 action id 映射与刷新策略尚未定义
- 当前稳定层中也尚未有相应设计
