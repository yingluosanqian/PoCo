# Decision

## 待选问题/方案

方案 A：新增一个 `/debug/env` HTTP 端点，返回 PoCo 进程当前可见的"关键环境变量"清单（只返回 key 是否存在和 length，不返回 value）。

方案 B：把环境变量可见性合并进 `/health` 响应，不另开端点。

方案 C：扩大到 `poco status` CLI 命令，也展示关键 env 列表。

方案 D：不做端点，直接给一份文档教用户用 `/proc/<pid>/environ`。

## 当前决策

采纳 **方案 A**：新增 `/debug/env`，独立端点，只暴露 key 存在性 + 值长度，不暴露 value。同时在 `/health` 的 `warnings` 里追加一行提示"详情见 /debug/env"。

不做 CLI（方案 C）这一轮，保持范围收敛。

## 为什么这样选

- **独立端点**：环境信息属于调试面，和健康探针定位不同。`/health` 会被 LB / 上游探活抓，混在一起会污染正常状态。
- **不暴露 value**：哪怕只是调试端点也不应该回传 `ANTHROPIC_API_KEY` 这种东西。返回 "存在 / 不存在 + 字符长度" 已经足够判断"有没有被继承到"。
- **白名单而非全量**：全量返回会泄漏无关信息，白名单覆盖当前四个 backend + 通用代理相关变量（HTTP_PROXY / HTTPS_PROXY / NO_PROXY）就够排障。
- **`/health` 只加一个提示**：让用户知道这个端点存在，不引入新字段。

## 为什么不选其他方案

- **方案 B**：会让 `/health` 响应结构膨胀，而且被上游探活工具抓到后可能被日志化，等于变相泄漏 env key 列表。
- **方案 C**：`poco status` 主要是面向本机终端用户的，而这类诊断经常发生在远程机器上。HTTP 端点更直接。CLI 支持可以留作后续小增量。
- **方案 D**：`/proc/<pid>/environ` 是 Linux only，在 darwin 上没有；而且 value 会直接暴露密钥，违反"最小泄漏"原则。

## 风险

- **白名单覆盖不全**：初版覆盖当前已知 backend + 代理相关，后续加 backend 时需要同步更新。
- **调试端点暴露面**：`/debug/env` 没有鉴权，和 `/debug/feishu` 同级。当前阶段 PoCo 还不开放鉴权体系；用户部署到公网前仍需自行加反代/鉴权。这条已经和既有 `/debug/feishu` 一致，不增加新风险。
- **用户可能误解"存在即可用"**：变量可能"被继承但值错"。需要在响应里明确说明这只是 "present" 判断，不保证 value 正确。

## 后续影响

- 新 backend 加入时，需要更新白名单。后续可以把白名单从 backend class 自己定义，main.py 再聚合。
- 后续若要做"把 `.bashrc` 变量持久化进 PoCo"的方案，本端点会变成验证入口。
- CLI 层（`poco status`）以后要支持时，后端已经有现成数据源，是小增量。
