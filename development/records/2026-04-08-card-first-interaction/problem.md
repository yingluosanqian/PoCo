# Problem

## 背景

PoCo 当前的实现仍围绕文本命令解析工作：无论是本地 demo 入口还是飞书消息入口，核心交互都依赖 `/run`、`/status`、`/approve`、`/reject` 这一类命令。

这与当前产品方向已经开始脱节。随着 project lifecycle、group workspace 和持续工作流变得更复杂，命令式交互会持续增加认知负担，并削弱手机端体验。

## 相关需求

- 用卡片 2.0 替代正式命令输入
- 在小屏场景下减少记忆和输入成本
- 用结构化交互承载 project 管理和任务执行

## 当前状态

- 当前正式实现仍以文本命令为核心交互协议
- 当前还没有卡片驱动的 project 管理和任务执行模型
- 当前命令式交互更像开发期 MVP，而不是面向正式用户的最终交互面

## 问题定义

PoCo 当前缺少一套以卡片 2.0 为中心的结构化交互模型，导致正式用户体验仍然依赖高认知负担的命令式交互，不符合移动端产品方向。

## 为什么这是个真实问题

- 小屏场景下，记忆和输入命令成本高，结构化卡片更符合项目目的里的“低摩擦交互”
- 随着 project、session、approval、group workspace 变复杂，命令输入会持续暴露上下文歧义
- 这已经不是“体验优化”，而是正式交互主路径与实现模型不一致

## 不是什么问题

- 不是“命令行解析器写得不够好”的问题
- 不是“现在立刻删除所有文本入口”的实现动作
- 不是“先做完整卡片设计系统”的问题

## 证据

- 发起人明确提出“不要用命令，全删了，直接用卡片 2.0 进行交互”
- 当前系统代码和文档仍把文本命令视为主交互协议
- 当前产品方向已转向 project lifecycle 和 workspace 模型，命令式交互负担会继续增大
