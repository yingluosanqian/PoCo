# Design

## 设计目标

把 PoCo 的正式交互从命令式文本协议转为卡片 2.0 驱动，同时保留最小开发后门，避免在实现过渡阶段失去验证与排障能力。

## 核心抽象

### 1. DM Management Cards

单聊中的 project 管理卡片。

职责：

- 展示 project 列表和概要
- 提供创建 project、建群、绑定等操作入口
- 承载低频配置动作

### 2. Group Workspace Cards

群聊中的 project 工作区卡片。

职责：

- 展示当前 session / task 状态
- 提供发起任务、确认、拒绝、查看结果等操作
- 取代主要文本命令入口

### 3. Fallback Command Path

过渡期的文本命令后门。

职责：

- 支持开发联调
- 支持卡片链路异常时的最小排障
- 不作为正式用户主路径继续扩张

## 模块/接口影响

- `interaction`
  未来需要从“文本命令解析”为中心转向“卡片 action 解析”为中心
- `platform/feishu`
  未来需要承接卡片交互和回调，而不只是文本消息事件
- `task/project/session`
  未来需要为卡片状态展示提供结构化视图模型
- `implementation`
  现有命令解析逻辑应从正式协议降级为 fallback

## 状态变化

当前：

- 文本消息进入解析器
- 解析器产生命令动作

后续建议：

1. 用户点击 DM 或群内卡片动作
2. 卡片 action 被解析为结构化意图
3. 意图路由到 project / session / task
4. 系统回写新的卡片视图与状态

## 方案比较

### 方案 A：命令主导，卡片辅助

优点：

- 对现有实现改动更小

缺点：

- 正式交互方向不一致

### 方案 B：卡片主导，命令 fallback

优点：

- 正式体验方向清晰
- 过渡风险可控

缺点：

- 需要维护双轨一段时间

### 方案 C：立刻删命令，只留卡片

优点：

- 路线最纯粹

缺点：

- 当前没有卡片实现，风险过高

## 最终方案

采用方案 B。

PoCo 的正式交互面转向卡片 2.0，现有文本命令仅作为过渡期 fallback 保留，不再被视为正式产品协议。

## 风险与兼容性

- 卡片 action 和状态更新需要额外回调链路支撑
- 过渡期若标识不清，用户和开发者可能误把 fallback 当正式入口
- 若飞书卡片能力存在限制，后续需要重新校准最小交互面
