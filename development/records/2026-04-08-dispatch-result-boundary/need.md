# Need

## 背景

当前已经定义了卡片最小信息架构、action intent、handler ownership 和 refresh mode，但业务层执行完一个 action 后，平台层究竟应该拿到什么结果结构，仍然没有被定义。

没有这一步，下一轮实现就会出现一个典型问题：handler 直接返回飞书卡片，或者平台层去猜业务结果，导致平台适配和核心逻辑重新耦合。

## 需求信号

- 需要继续推进实现前设计
- 需要把 dispatcher 返回结构定下来
- 需要把业务层和平台渲染层的边界钉死

## 来源

- 当前设计已经进入最后一层实现前收敛

## 场景

- handler 执行 `task.approve` 后，平台层需要知道是追加结果卡还是仅回执
- handler 执行 `project.open` 后，平台层需要知道要替换当前卡并展示哪种 view
- 相同业务结果可能在 DM 和群里呈现不同卡片

## 频率/影响

这一步将直接决定实现层是否还能保持“薄平台适配 + 独立任务核心”的结构。

## 备注

本轮仍然不落到飞书 schema 级别，而是定义平台无关的返回结构。
