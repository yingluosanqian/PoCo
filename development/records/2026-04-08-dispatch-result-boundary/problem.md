# Problem

## 背景

PoCo 已经定义了“卡片动作如何进入后端”，但还没有定义“后端处理完之后如何把结果交还给平台层”。

若这一层缺失，后续实现会在两个坏方向里二选一：

- handler 直接拼飞书卡片
- 平台层读取业务对象并自行猜测如何渲染

## 相关需求

- 业务层返回稳定结果
- 平台层按稳定指令渲染卡片
- 同一业务结果可以适配不同平台表面

## 当前状态

- 当前 records 只定义到 `ActionIntent -> handler`
- 当前没有 dispatcher 返回结构
- 当前没有“业务 view model”和“平台渲染指令”的分层定义

## 问题定义

PoCo 当前缺少一套平台无关的 dispatch result 边界，导致 card-first 实现仍可能在业务层和平台渲染层之间重新耦合。

## 为什么这是个真实问题

- 这会直接破坏目前反复强调的“薄平台适配”方向
- 这会让飞书卡片 schema 细节渗透进核心逻辑
- 以后接其他平台时会非常痛苦

## 不是什么问题

- 不是最终模板语法问题
- 不是飞书卡片 JSON 长什么样的问题
- 不是立刻写 renderer 代码的问题

## 证据

- 当前 records 尚未定义 dispatch result 结构
- 当前稳定层虽然强调平台适配应尽量薄，但还没有具体边界对象
