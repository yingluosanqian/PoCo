# Problem

PoCo 当前把“群里发任务”主要收敛到 `Run Task -> task_composer -> submit` 这条 card-first 路径上。

这条路径虽然结构清楚，但在移动端高频使用场景里交互成本偏高：

- 发起动作需要先找卡片
- 需要二次点击进入输入框
- 需要在卡片内再次输入

这使群工作区不像“和 agent 对话”，更像“操作一个表单系统”。

真正的问题不是“有没有 task_composer 卡片”，而是：

群工作区的默认输入语义还没有切换成对话式 prompt intake，导致正式工作流虽然可用，但不够自然。

这不是底层任务执行模型的问题，也不是 card-first 整体方向失效的问题。

它是一个入口语义问题：

- 群消息默认应该被当成什么
- 文本消息与卡片动作如何分工
- DM 和 Group 的默认语义是否一致
