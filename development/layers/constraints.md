# Constraints Layer

## 本层定义

本层定义为了实现项目目的，哪些东西不能牺牲，哪些路径不能接受。

本层是护栏，不是建议。

## 本层关注的问题

- 哪些风险不能接受
- 哪些复杂度不能引入
- 哪些代价不能支付
- 人和 AI 在实现时必须遵守哪些边界

## 输入

- `purpose`
- 历史失败经验
- 安全、质量、复杂度和维护成本要求
- 运行环境和协作约束

## 输出

- 风险边界
- 复杂度边界
- 演进护栏
- 明确的不接受路径

## 禁止事项

- 不要把局部偏好写成硬约束
- 不要把暂时实现不了的内容伪装成约束
- 不要允许实现阶段绕开约束
- 不要出现“特殊情况先做了再说”的默认口子

## 进入下一层的条件

满足以下条件后，才能进入问题定义和决策：

- 能明确说明什么不能被牺牲
- 能识别哪些方案虽然可行但不可接受
- 后续层可以用本层排除错误路径

## 当前最小可用版本

第一版至少应明确：

- 不可接受的风险
- 不可接受的复杂度增长
- 不允许越过的实现边界

## PoCo 当前项目约束

### 安全边界

- 不允许把服务器上的 agent 控制能力直接暴露给未鉴权的消息入口
- 不允许默认信任任意来自聊天平台的事件、用户身份或回调请求
- 不允许让高风险操作在没有明确确认的情况下直接执行
- 不允许把密钥、令牌、会话凭证或敏感执行结果泄露到普通消息流中
- 不允许因为追求移动端便利而绕过审计、权限控制或最小授权原则

### 复杂度边界

- 第一阶段不接受为了“以后也许会支持很多平台”而提前做重型抽象
- 第一阶段不接受把系统做成通用 bot 平台、通用工作流引擎或通用多租户控制台
- 第一阶段不接受同时解决消息接入、agent 编排、权限系统、结算系统和完整协作体系
- 不允许在目标尚未验证前引入过多后台服务、异步链路或难以维护的状态机
- 不允许为了形式完备而显著拖慢可交付速度

### 交互边界

- 移动端交互必须围绕“发起任务、查看进展、处理确认、接收结果”四类核心动作展开
- 不接受把手机端交互设计成需要长时间编辑、复杂配置或高密度信息阅读的界面
- 不接受高噪声通知；默认应压缩无价值过程消息，只保留能帮助判断和决策的信息
- 需要人工确认的节点必须清晰，不能让用户在手机上做模糊授权
- 交互设计必须适应碎片化使用，允许用户中断后恢复上下文

### 平台策略边界

- 飞书 bot、Slack bot 或类似消息入口当前可以作为优先接入方向，但不应反向定义产品目的
- 第一阶段不要求同时支持多个消息平台
- 不允许为了平台兼容性牺牲核心工作流清晰度
- 平台适配应尽量薄，核心能力应尽量留在平台无关的一侧

### 执行模型边界

- 核心执行必须以服务器侧 agent 为中心，而不是依赖手机本地常驻、长连接或本地算力
- 用户离开聊天窗口后，任务仍应能继续推进或安全暂停，而不是整体失效
- 不允许把“远程 shell”直接伪装成“agent 控制”；系统应围绕任务和状态，而不是裸命令透传
- 不允许在没有状态可见性的情况下启动长任务

### 产品范围边界

- 当前不以支持团队级复杂协作、多人并发指挥同一 agent 为前提
- 当前不以替代桌面开发环境为目标
- 当前不以覆盖所有开发场景为目标，应先聚焦少数高价值、可重复的移动控制场景
- 当前不承诺通用插件生态、开放平台或复杂扩展机制

### 当前阶段的硬性取舍

在早期阶段，PoCo 应优先保证以下顺序：

- 先保证安全可控，再追求操作自由度
- 先保证单条主链路跑通，再追求平台数量和功能广度
- 先保证任务状态清晰，再追求交互花样和自动化程度
- 先保证真实可用，再追求架构上的泛化与完备
