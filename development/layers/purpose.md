# Purpose Layer

## 本层定义

本层定义项目为什么存在，以及项目持续演进时优先优化什么。

它不是需求池，也不是愿望清单。它只负责给后续所有选择提供判断基准。

## 本层关注的问题

- 项目存在的核心理由是什么
- 系统要持续优化什么
- 什么结果算成功
- 哪些事情虽然合理，但不属于本项目目的

## 输入

- 项目发起背景
- 产品方向
- 长期用户价值判断
- 组织或维护者的明确目标

## 输出

- 一组稳定、可复用的项目目标表述
- 一组成功判断标准
- 一组非目标边界

## 禁止事项

- 不要把具体需求写成本层内容
- 不要把当前实现方案写成本层内容
- 不要把“什么都想做”写成目的
- 不要为了局部请求临时改写项目目的

## 进入下一层的条件

满足以下条件后，才能进入约束层和问题层：

- 项目存在理由可被简洁复述
- 成功标准足够清楚，可用于排除低价值工作
- 非目标边界已经明确

## 当前最小可用版本

若项目尚未形成完整产品定义，第一版至少应明确：

- 项目服务什么类型的问题
- 项目优先优化什么
- 项目明确不负责什么

## PoCo 当前项目目的

### 项目名称

PoCo = Pocket Coding

### 项目存在理由

PoCo 的存在，是为了让开发者不必被电脑和固定工作台绑住，也能通过手机持续控制运行在服务器上的 AI agent。

它要解决的核心问题不是“在手机上写完整代码”，而是“在移动场景下，仍然能以足够低的摩擦发起任务、查看进展、给出确认、接收结果，并让远端 agent 持续工作”。

### 优先优化的结果

- 在手机上可以用简洁方式驱动远端 AI agent 完成真实开发任务
- 用户可以在消息入口中查看上下文、关键进展、阻塞点和结果
- 用户可以在移动端完成必要确认，而不必回到桌面端才能继续推进
- 远端执行与移动端交互解耦，agent 可以在服务器上持续运行
- 交互成本足够低，适合碎片化、随时打断又随时恢复的使用方式

### 成功判断标准

若 PoCo 有效，至少应满足以下判断：

- 用户可以仅通过手机发起并推进一次真实的 agent 工作流
- 用户可以在手机上获知 agent 当前在做什么，而不是只收到最终结果
- 当 agent 需要确认、授权或补充信息时，用户可以在移动场景下及时处理
- 任务执行主要依赖服务器侧能力，而不是依赖手机本地算力或长时在线
- 消息交互足够清晰，避免在小屏场景下产生大量低价值噪声

### 非目标边界

当前 PoCo 明确不以以下内容为核心目标：

- 不把手机变成完整 IDE
- 不追求在手机本地运行重型 AI agent 或开发环境
- 不以支持尽可能多的平台接入为第一目标
- 不把聊天入口本身做成复杂协作平台
- 不以一次性 demo 或玩具级自动化为目标，而应服务真实可持续的远端开发工作流

### 当前方向约束

现阶段可以把飞书 bot、Slack bot 或类似消息入口视为高优先级交互方式，但它们是实现路径，不是项目目的本身。
