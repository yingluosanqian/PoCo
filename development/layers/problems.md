# Problems Layer

## 本层定义

本层负责把 `purpose + constraints + needs + state` 收敛成真正值得定义的问题。

问题不是需求原文，也不是实现方案。

## 本层关注的问题

- 当前最值得解决的矛盾是什么
- 这个矛盾为什么会阻碍项目目的
- 它与需求信号的关系是什么
- 它不是什么问题

## 输入

- `purpose`
- `constraints`
- `needs`
- `state`

## 输出

- 清晰的问题定义
- 问题边界
- 问题成立的证据
- 不属于该问题的内容

## 禁止事项

- 不要把需求句子直接复制为问题定义
- 不要把方案写成问题
- 不要同时把多个不相干问题揉成一个问题
- 不要在证据不足时强行立题

## 进入下一层的条件

满足以下条件后，才能进入决策层：

- 问题定义能说明“为什么现在值得解决”
- 问题边界清楚
- 已明确“不是什么问题”
- 有足够证据支撑其真实性

## 问题和需求的区别

- 需求回答“外部在表达什么”
- 问题回答“系统当前真正阻碍目的的是什么”

需求可以很多，问题应尽量少而清楚。

## 问题记录最少应包含

- 背景
- 相关需求
- 当前状态
- 问题定义
- 为什么是真问题
- 不是什么问题
- 证据

## PoCo 当前问题

PoCo 当前最值得优先解决的问题是：

开发者在移动场景下，缺少一个低摩擦、可控、可持续的方式，去发起、跟进并必要时确认服务器侧 AI agent 的真实工作流。

### 为什么这是真问题

- 这直接阻碍了 PoCo 的项目目的成立
- 当前完全不存在可运行主链路，而不是只缺一个局部功能
- 这是后续消息入口、任务模型和确认机制决策的共同上游问题

### 不是什么问题

- 不是“先做哪个平台接入”本身的问题
- 不是“在手机上提供完整编码环境”的问题
- 不是“做通用聊天机器人平台”的问题

### 当前问题边界

这个问题关注的是“移动场景下如何有效控制远端 agent 工作流”，而不是所有潜在的协作、平台扩展或桌面替代需求。

### 当前新暴露的次级问题

随着飞书入口、任务执行和状态回推已经出现，PoCo 下一步新暴露出的关键问题是：

系统还缺少产品级连续性交接上下文，导致用户难以在移动场景下低摩擦恢复和延续同一条工作流。

同时，系统也缺少清晰的 project 级消息容器模型，导致多 project 并存时，移动端交互容易出现归属混乱。
