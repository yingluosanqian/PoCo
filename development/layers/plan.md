# Plan Layer

## 本层定义

本层负责把已经被选中的方向变成一轮可执行变更。

它定义范围、风险、验收标准和实施顺序。

## 本层关注的问题

- 本轮到底改什么
- 本轮明确不改什么
- 主要风险是什么
- 如何判断这一轮完成

## 输入

- 决策记录
- 当前状态
- 资源和时间预算
- 相关依赖条件

## 输出

- 变更范围
- 不在范围内的内容
- 风险点
- 验收标准
- 实施顺序

## 禁止事项

- 不要在没有计划时直接做跨模块实现
- 不要把愿望列表写成范围
- 不要缺失“不在范围内的内容”
- 不要把测试动作等同于验收标准

## 进入下一层的条件

满足以下条件后，才能进入设计或实现：

- 范围已经清楚
- 风险已被显式写出
- 验收标准足够判断完成与否
- 实施顺序可支持逐步落地

## 为什么实现前必须有计划

因为没有计划时，实现最容易做三件坏事：

- 擅自扩大范围
- 漏掉关键风险
- 完成后无法判断是否真正交付

## PoCo 当前计划方向

### 当前轮目标

围绕“飞书优先”的单平台 MVP，定义一条可执行的最小主链路。

### 当前轮范围

- 明确用户如何从手机发起远端 agent 任务
- 明确任务进展如何回传到消息入口
- 明确哪些节点需要人工确认
- 明确服务器侧执行与消息入口之间的边界

### 当前轮不在范围内

- Slack 或其他平台同步接入
- 团队级协作与多租户能力
- 通用 bot 平台化
- 手机端完整 IDE 式体验

### 当前轮验收导向

当本轮结束时，应该能清楚回答：

- 第一条主链路是什么
- 为什么先做飞书
- 本轮不做什么
- 下一步设计和实现的边界是什么

### 当前下一轮计划方向

在主链路已基本跑通后，下一轮计划应优先定义最小 `session/handoff` 层，而不是继续把所有连续性压力压到 task 或 backend 内部上下文上。

同时，需要补出 `DM -> project lifecycle -> group workspace -> session -> task` 的正式交互模型，再进入具体实现。

在交互协议已经转向 card-first 后，下一轮实现前设计应先围绕 DM 管理卡片和群工作区卡片的最小信息架构展开。

在群工作区首卡已经落地后，下一轮设计与实现应优先围绕执行上下文配置展开：

- 在 DM 中定义 project 级 `agent` 选择与默认 workdir 配置入口
- 在群中定义 session 级 `working dir` 切换入口
- 明确 `task` 只消费当前上下文，不再承担重配置职责

在进入实现前，还应继续把这两类入口细化为最小卡片 IA：

- DM 的 `Project Config Card`
- 群里的 `Workdir Switcher Card`

在群正式工作流上，下一轮实现应优先调整默认消息语义：

- 已绑定 project 的群里，普通文本消息直接创建 task
- DM 继续保持 control plane 语义
- `Run Task` 卡片收缩为辅助输入入口，而不是唯一正式入口
